# OpenAI LLM Data Policy Notes

**Status:** Active
**Type:** Research
**Audience:** Both
**Date:** 2026-04-28

## Summary

This note summarizes the public OpenAI data-use, training, retention, and subprocessor policy surface relevant to using OpenAI LLMs in Parley through OpenAI API endpoints. It is written for people who want to verify that Parley is not accidentally using their words, ideas, confidential material, or other intellectual property in a way that gives away their IP or trains a third-party model.

This is an engineering policy note, not legal advice. It records the public policy surface Parley relies on as of 2026-04-28.

Parley's intended OpenAI usage relies on OpenAI's API Platform and business/developer policy surface, not consumer ChatGPT. Under OpenAI's public API data controls, model-training policy, services agreement, and data processing addendum:

- OpenAI says data sent to the OpenAI API is not used to train or improve OpenAI models unless the customer explicitly opts in to share data.
- OpenAI says organizations using business products, including the API Platform, are opted out of data sharing by default.
- OpenAI says it will not use Customer Content to develop or improve the Services unless the customer explicitly agrees to that use.
- OpenAI says API abuse monitoring logs are generated by default and retained for up to 30 days, unless longer retention is required by law or reasonably necessary to protect OpenAI's services or a third party from harm.
- OpenAI offers Modified Abuse Monitoring and Zero Data Retention controls to eligible approved customers.

Based on those public statements, OpenAI appears to satisfy Parley's current LLM privacy target for ordinary stateless API inference: no model training by default and ordinary abuse-log retention under 90 days.

Important caveats:

- Public policy pages can change. Re-verify before making contractual promises.
- Abuse, safety, legal, compliance, and service-protection exceptions may allow review or longer retention.
- OpenAI endpoint behavior is not uniform. Some API features store application state beyond abuse logs, especially if data is intentionally stored by the customer or if Parley uses persistent endpoints.
- For `/v1/responses`, OpenAI documents 30-day application-state retention by default or when `store` is `true`; when Zero Data Retention is enabled, `store` is treated as `false`.
- Persistent objects such as assistants, threads, vector stores, files, fine-tuning jobs, evals, and batches have endpoint-specific storage and deletion behavior. Parley should avoid those features for privacy-sensitive LLM use unless separately disclosed.
- Image and file inputs may have special safety scanning and manual-review retention exceptions.
- Local Parley storage is separate from OpenAI's policy. If Parley stores prompts, model outputs, transcripts, or generated artifacts locally, that local data must be governed by Parley's own deletion and access controls.

## Subprocessor Disclosure

Parley may send user-provided text, transcript excerpts, conversation context, prompts, and related metadata to OpenAI as a third-party LLM API provider. OpenAI processes that data to provide model inference and related API services. OpenAI maintains a public subprocessor list for services covered by its Data Processing Addendum.
